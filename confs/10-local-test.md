# Local Test Harness (`local_test/`)

A standalone replica of the production pipeline that runs entirely on a laptop ‚Äî **no GCP, no GPU, no Pi**. Used for iterating on judge logic, alert behaviour, or just sanity-checking a model with a webcam.

## What it replaces

| Production component | Local replacement | File |
|---|---|---|
| Pub/Sub `frames-in` | Direct HTTP POST | `orchestrator.py` |
| Dispatcher Cloud Function | Eliminated ‚Äî orchestrator POSTs directly to judge | ‚Äî |
| Vertex AI endpoint | `HTTPServer` on `localhost:8080` | `local_judge.py` |
| YOLOv8x on T4 GPU | Same `best.pt`, CPU inference (`_model.to("cpu")`) | `local_judge.py` |
| Pub/Sub `detections-out` | Returned in HTTP response body | `orchestrator.py` |
| Firestore `alerts` collection | In-memory list `_alert_log` | `local_alert_handler.py` |
| Firestore `alert_cooldowns` | In-memory dict `_cooldowns` | `local_alert_handler.py` |
| Gmail SMTP | **Same code** ‚Äî skipped if creds missing | `local_alert_handler.py` |
| FCM push | Omitted (console log only) | ‚Äî |
| Raspberry Pi camera (RTSP) | Local webcam / video / image dir / single image | `orchestrator.py` |

## Source mapping (production ‚Üî local)

The local files are *near-copies* of the production sources, edits restricted to:
- `local_judge.py` ‚âà `terraform_v2/services/judge/main.py` **minus** the `google.cloud.pubsub_v1` import / publish; **plus** a `to("cpu")` call.
- `local_alert_handler.py` ‚âà `terraform_v2/services/alert-manager/main.py` **minus** `google.cloud.firestore` and `functions_framework` decorators; **plus** in-memory cooldown dict + markdown-row file logging.

Same `HIGH_SEV = {"spagetti", "not_sticking", "layer_shift", "warping"}`, same `CONF_THRESHOLD=0.35` default, same `COOLDOWN_SECONDS=60` default (the local default is 60 s; the production default is 300 s ‚Äî `local_alert_handler.py:19`).

## Usage

```bash
cd local_test

# Webcam 0, 1 fps (defaults)
./run_local.sh

# Explicit webcam at 2 fps
./run_local.sh --source 0 --fps 2.0

# Three webcams in parallel (indices 0, 1, 2 ‚Üí cam1, cam2, cam3)
./run_local.sh --source 0 --cameras 3

# Video file
./run_local.sh --source /path/to/print.mp4

# Image directory (loops)
./run_local.sh --source /path/to/frames/

# Single image (loops forever ‚Äî useful for verifying that a known defect fires)
./run_local.sh --source /path/to/spaghetti.jpg

# Custom model or conf
./run_local.sh --source 0 --model /other/best.pt --conf 0.4

# Send real emails on HIGH_SEV (uses prod-style creds)
GMAIL_ADDRESS=you@gmail.com GMAIL_APP_PASSWORD="xxxx xxxx xxxx xxxx" ./run_local.sh
```

## Argument and env-var reference

`orchestrator.py` CLI:
| Flag | Default | Meaning |
|---|---|---|
| `--source` | `0` | webcam index, video file, image dir, single image |
| `--cameras` | `1` | when source is int, number of consecutive webcam indices to spawn |
| `--camera-id` | `cam1` | label used for single-source modes |
| `--fps` | `1.0` | capture / playback rate |
| `--model` | unset | overrides `MODEL_PATH` env var |
| `--conf` | unset | overrides `CONF_THRESHOLD` env var |

Env vars (read by `local_judge.py` and `local_alert_handler.py`):
| Var | Default |
|---|---|
| `MODEL_PATH` | `../terraform_v2/services/judge/best.pt` |
| `CONF_THRESHOLD` | `0.35` |
| `COOLDOWN_SECONDS` | `60` (note: prod is 300) |
| `GMAIL_ADDRESS` | `""` (email skipped if empty) |
| `GMAIL_APP_PASSWORD` | `""` |

`run_local.sh` sets the first three to sensible defaults before invoking `orchestrator.py`. Args after the script name are forwarded.

## Logging

Each session appends a **markdown table** to `local_test.md`. The header is generated by `_log_session_header()` in `orchestrator.py:89-98`:

```markdown
## Session ‚Äî Mon 04 May 2026, 14:25:52 UTC

- **Source:** `0`
- **FPS:** 2.0
- **Model:** `../terraform_v2/services/judge/best.pt`
- **Confidence threshold:** 0.35

| Time | Camera | Seq | Detections |
|------|--------|-----|------------|
| Mon 04 May 2026, 16:25:53 | cam1 | 0 | spagetti (87%) |
| ...
```

Each detection (confidence ‚â• threshold) appends one row. `human_time()` converts ISO timestamps to `Europe/Bucharest` local time.

## How the orchestrator runs

`orchestrator.py:219-253`:
1. Parse args, apply `--model` / `--conf` to env.
2. `_start_judge()` ‚Äî imports `local_judge`, calls `init()` (loads YOLO), starts the HTTP server in a daemon thread, polls `/healthz` for up to 20 s.
3. `_log_session_header()` writes the markdown stanza.
4. Branch on the source type:
   - **int** + `cameras > 1` ‚Üí `_run_webcam_multi(N, fps)` ‚Äî N daemon threads, each with `_run_webcam(i, "cam{i+1}", fps)`.
   - **int** + `cameras == 1` ‚Üí `_run_webcam(device, camera_id, fps)`.
   - **dir** ‚Üí `_run_image_dir(dir, camera_id, fps)` ‚Äî loops indefinitely.
   - **video file** (`.mp4`/`.avi`/`.mov`/`.mkv`/`.webm`) ‚Üí `_run_video(path, camera_id, fps)` ‚Äî loops on EOF.
   - **image file** (`.jpg`/`.jpeg`/`.png`/`.bmp`) ‚Üí `_run_single_image(path, camera_id, fps)` ‚Äî re-encodes the same frame forever.
5. Each capture loop: read frame ‚Üí `_encode_frame()` (JPEG q=70 ‚Üí base64) ‚Üí `_post_and_handle()` (POST to local judge, then call `local_alert_handler.handle_detection(result)`).

## Differences from production (cheat sheet)

- **No Pub/Sub** ‚Äî judge returns predictions inline, orchestrator passes them straight to the alert handler.
- **Cooldown TTL**: local default 60 s vs prod 300 s.
- **No Firestore** ‚Äî cooldowns are in-process; lost on restart.
- **No FCM** ‚Äî push notifications aren't simulated.
- **No streak filter** ‚Äî `local_judge.py` does **not** implement the per-camera streak logic; it emits every detection that beats the conf threshold. This is intentional for local debugging.
- **Email subject** is prefixed `[LOCAL TEST]` so test emails are visually distinct in your inbox.
- **CPU inference** ‚Äî `_model.to("cpu")` in `local_judge.py:37`. Inference is ~5-10√ó slower than T4; at 1 fps you won't notice.

## Why this isn't in CI

It needs a webcam / video file / model weights / OS-level OpenCV ‚Äî too heavy for the GitHub Actions runner. Unit tests in [[07-testing]] cover the pure logic; this harness covers integration up to the GCP boundary.

If you want to add it to CI, the smallest useful test is single-image looping (`./run_local.sh --source <known_defect.jpg>` for 5 seconds, check `local_test.md` got the row).
